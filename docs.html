EnvVault

"Git for environment variables." A complete build specification — product scope, architecture, data model, API contract, CLI spec, and security model — written so implementation can start without a kickoff meeting.

01 Overview

the problem, the vision, who this is for

The problem

Every codebase depends on secrets — database URLs, API keys, JWT secrets, payment credentials. They live in .env files kept out of Git via .gitignore. That protects them from source control, but creates a second, unsolved problem: developers lose local .env files, teammates share secrets over Slack/WhatsApp, nobody knows which copy is current, and there's no record of who changed what.

The vision

EnvVault makes encrypted secret sharing as simple as source control already is — reusing the exact mental model developers already have: init, pull, push. Nothing new to learn. No developer should ever again need to ask "can someone send me the latest .env?"

Who it's for

Primary

Solo developers, freelancers, 2–20 person startup teams, agencies, open-source maintainers, student teams.

Deliberately not

Large enterprises with dedicated platform/security teams — that segment is already served (and over-served) by Vault, Infisical Enterprise, and CyberArk.

Core principles

  • Simple — if it needs a manual, it's already too complex.
  • Secure by default — encryption is not a toggle, it is the only mode.
  • Git-shaped — reuse the mental model developers already have.
  • Fast — first working pull in under 2 minutes from signup.
  • Browser-safe — full encryption with no desktop app requirement.

🔒

Non-negotiable rule: Express and MongoDB only ever see and store ciphertext. All encryption/decryption happens client-side — in the React app via WebCrypto, and in the CLI via Node's crypto module — using a per-project key that is itself wrapped per-user and never persisted in plaintext server-side. This is what lets us claim true end-to-end encryption without forcing a desktop app install, unlike EnvKey.

04 Data Model

MongoDB collections, exact fields, relationships

schema.js
users
  _id, oauthProvider, oauthId, email, name, avatarUrl, createdAt

projects
  _id, name, ownerId, members: [{ userId, role }], createdAt, archivedAt

environments
  _id, projectId, name // "development" | "staging" | "production" | custom, createdAt

variables
  _id, environmentId, key, encryptedValue, iv, updatedBy, updatedAt

activityLogs
  _id, projectId, actorId, action, targetType, targetId, metadata, createdAt

encryptionKeys
  _id, projectId, userId, encryptedProjectKey // project key wrapped per-user

variables.encryptedValue is ciphertext only — Express and MongoDB never see or store a plaintext secret at any point in its lifecycle.

Indexes required

  • projects: index on ownerId, and on members.userId
  • environments: compound index on (projectId, name), unique
  • variables: compound index on (environmentId, key), unique
  • activityLogs: index on (projectId, createdAt desc) for feed pagination

05 Security Model

the encryption design that is our core differentiator — implement exactly as specified

Key hierarchy

  1. On project creation, the client generates a random project key (AES-256) — never sent to the server in plaintext.
  2. The project key is wrapped ("encrypted") separately for each member using a key derived from that member's session, and stored as encryptedProjectKey in the encryptionKeys collection.
  3. Every variable value is encrypted client-side with AES-256-GCM using the unwrapped project key, producing encryptedValue + iv.
  4. When a member is added to a project, the client re-wraps the project key for that new member — the server relays this wrapped blob but cannot read the underlying key.

What the server can and cannot do

Server can

Store ciphertext, route wrapped keys between authorized members, log metadata (who/when/what action), enforce access control on which projects a user can reach.

Server cannot

Read any secret value in plaintext, at rest or in transit — a full database compromise alone does not expose usable secrets.

Other required controls

  • HTTPS-only transport, HSTS enabled
  • Secrets hidden by default in the UI; "reveal" is an explicit, logged, client-side-only decrypt action
  • Mandatory extra confirmation step before modifying or deleting anything inside a production environment
  • JWT session tokens with short expiry + refresh rotation
  • Full activity logging for every sensitive action (see Activity module)
  • OAuth-only authentication — removes password storage/reset as an attack surface entirely
⚠

Do not build any server-side "decrypt for support/debugging" pathway. If it exists, the end-to-end claim is false. If a user loses all their devices and local key material, the correct behavior is guided key re-wrap by remaining project members — not a server-side master key.

06 Authentication

OAuth-only — deliberately no password flows in MVP

Login is GitHub or Google OAuth only. There is no password field, no "forgot password" flow, and no password-reset email pipeline to build, maintain, or secure. This removes the single largest source of account-related support tickets and vulnerabilities in comparable products.

User clicks "Continue with GitHub" →
OAuth redirect + consent →
Express exchanges code for provider profile →
User record created/matched by oauthId →
JWT session issued, stored in httpOnly cookie

CLI auth uses a device-code flow (see CLI spec) so a headless terminal session can still authenticate through the same OAuth providers.

07 Modules & Features

every module required for MVP, in build order

1. Projects

Top-level container. Holds environments, members, and activity.

Features: create · rename · delete · archive · project dashboard.

2. Environments

Behave like Git branches — same command shape, zero new concepts to learn.

Features: create · rename · delete · clone (copy one environment's variables into a new one).

3. Variables

Key-value secrets inside an environment, always encrypted client-side.

Features: add · edit · delete · search · copy value · hide/reveal (client-side decrypt only) · bulk import from a dropped .env file (auto-detects target environment from filename, e.g. .env.production) · bulk export.

4. Environment Comparison — MVP, not post-launch

Side-by-side diff between any two environments in a project.

Highlights: missing keys · changed values (without revealing plaintext unless explicitly unlocked) · extra/unused variables. This is pulled forward into MVP deliberately — it is the single most-cited daily-value feature developers look for, and a known gap in Doppler's free tier.

5. CLI

The primary day-to-day interface — see full spec in section 09.

6. Team Management

Kept intentionally minimal for MVP.

Features: invite member (by email, resolved to their OAuth account on signup) · remove member · view members · transfer ownership. Granular role-based permissions are explicitly out of scope for MVP — a small team needs "owner" and "member," not a permissions matrix.

7. Activity Log

One unified feed per project, not a fragmented multi-screen log.

Tracked events: variable added/updated/deleted · environment created/deleted · member invited/removed · push/pull executed.

09 CLI Specification

the primary day-to-day developer interface — must work via npx, no global install required

terminal
# authenticate — opens a device-code OAuth flow in the browser
npx envvault login

# link the current folder to a project + environment
npx envvault init

# decrypt remote secrets and write a local .env
npx envvault pull

# diff local vs remote, confirm, encrypt, upload
npx envvault push

# show drift between local .env and remote state
npx envvault status

envvault push — exact required behavior

Read local .env →
Detect changes vs. last known remote state →
Print a plain-language diff (+added / ~changed / -removed keys, no values shown for unchanged keys) →
Prompt for confirmation — require a second, explicit confirmation if target environment name is "production" →
Encrypt changed values client-side (AES-256-GCM) →
Upload ciphertext only via POST /cli/push

Local config file

Written by envvault init to .envvault/config.json (must be added to the project's .gitignore automatically by the CLI):

.envvault/config.json
{
  "projectId": "proj_9f2a",
  "environment": "development",
  "apiBase": "https://api.envvault.dev"
}

12 MVP Scope

what ships in v1 — no negotiation on the "excluded" list without revisiting section 02

Included

OAuth authentication (GitHub, Google) · Projects (create/rename/delete/archive) · Environments (create/rename/delete/clone) · Variables (add/edit/delete/search/bulk import-export/hide-reveal) · Environment comparison/diff · Client-side end-to-end encryption · CLI (login, init, pull, push, status) via npx · Basic team management (invite/remove/owner transfer) · Unified activity log.

Excluded from MVP

GitHub Actions/CI integrations · VS Code extension · Secret version rollback/full history · Granular role-based permissions · Secret rotation automation · Public API/SDK for external integrations · Billing/subscriptions · PKI, certificate management, dynamic secrets, agent-credential governance.

14 Future Enhancements

everything planned beyond MVP — what it is, why it matters competitively, and how it would be built

None of these ship in v1. They exist here so the team has a shared understanding of where the product goes next, and so MVP decisions (data model, API shapes, crypto design) are made in ways that don't have to be undone later. Each item below lists what it is, why it matters, and how it would be implemented.

↺ Version History & Rollback
Phase 2

Every push to an environment is currently overwrite-only. This turns each environment into a timeline: every push creates an immutable version snapshot, viewable and restorable.

What

A version list per environment (v1, v2, v3…), each showing which keys changed, who changed them, and when. One click restores any prior version.

Why

Someone will eventually push a bad value to production. Right now that's unrecoverable without asking a teammate — this makes it a 10-second self-service fix, and it's the natural next step once diff/compare (already in MVP) is trusted.

How

New versions collection — append-only, one document per push, storing the full encrypted key/value set at that point in time. Restoring re-encrypts nothing; it just re-points the environment's "current" pointer.

⌘ VS Code Extension
Phase 2

Pull, push, and browse secrets without leaving the editor.

What

A sidebar panel listing variables (masked by default), inline pull/push commands, and a status-bar indicator showing whether the local .env is in sync with the remote environment.

Why

Developers shouldn't need to tab out to a browser or terminal for something this routine. This is where daily habit gets locked in — and neither Doppler nor EnvKey have a polished version of this today.

How

Built on the VS Code Extension API, reusing packages/crypto-core directly (same encrypt/decrypt logic as web and CLI — no third implementation to keep in sync).

⚙ GitHub Actions / CI Integration
Phase 3

Extend "push once, sync everywhere" from local development into deployment pipelines.

What

An official GitHub Action that injects a decrypted environment as CI/CD env vars at pipeline runtime, with values automatically masked in logs.

Why

Removes the last place teams still paste secrets manually — CI/CD platform settings — and closes the gap with Infisical/Vault on this specific workflow without adopting their full complexity.

How

A scoped, short-lived CI token (not a user's personal OAuth session) authenticates the Action; it calls GET /cli/pull server-side and exports decrypted values only into the ephemeral runner environment.

{ } Public API & Language SDKs
Phase 3

Let applications fetch their own secrets at runtime instead of relying solely on a generated .env file.

What

Project-scoped API keys, a documented REST API, and thin SDKs (Node, Python, Go first) for fetching and decrypting secrets programmatically.

Why

Some teams want runtime secret injection (12-factor style) rather than a generated file — this is the natural on-ramp without forcing every team into it.

How

API keys are scoped to a single project + environment and can be revoked independently of any user's session; SDKs wrap the existing /cli/pull-equivalent endpoint and perform decryption client-side, same as the CLI.

☰ Role-Based Permissions
Phase 4

Grows the MVP's simple owner/member model only once teams actually need it.

What

Roles like Admin, Developer, and Viewer, with per-environment scoping — e.g. a contractor gets read/write on "development" only, never "production."

Why

Deliberately not in MVP — a 3-person team doesn't need a permissions matrix, and building it early would add UI and decision-making overhead for exactly the audience we're trying to stay simple for. Add it only when usage data shows teams outgrowing owner/member.

How

Extend projects.members with a role and optional scopedEnvironments array; enforce in Express middleware, not in the client.

↻ Secret Rotation Scheduling
Phase 4

Automatically rotate selected secrets (e.g. database credentials) on a schedule or on demand.

What

A rotation policy per variable (e.g. "every 30 days"), with the new value generated, encrypted, and pushed automatically, and the old one invalidated at the provider.

Why

This is the first feature that starts to resemble Vault/Infisical territory — it's placed last and explicitly optional so EnvVault doesn't chase compliance-driven complexity before the core product has proven itself.

How

Provider-specific rotation adapters (starting with common databases), triggered by a scheduled job that calls the provider's API, then performs the same client-side-encrypt-and-push flow the CLI uses.

⌂ Self-Hostable Open-Core Server
Phase 4

A Docker-deployable version of the API + database for teams that need full infrastructure control.

What

An open-source core (Express API + schema) that a team can run on their own infrastructure, pointed at their own MongoDB instance, with the hosted EnvVault remaining the default, easier path.

Why

Matches Infisical's trust story for compliance-sensitive teams without making self-hosting the default experience — most of our audience wants zero infrastructure to manage, so this stays opt-in.

How

Because the encryption boundary already lives entirely client-side, self-hosting the server changes nothing about the security model — it only changes who operates the ciphertext store.

✉ Notifications
Phase 3

Keep the team aware of sensitive changes without anyone needing to check the activity log manually.

What

Slack, Discord, and email alerts for production variable changes, new members joining/leaving, and new environments being created.

Why

Turns the existing activity log from a "pull" experience (someone has to go look) into a "push" experience (the team finds out immediately) — cheap to build, high perceived value.

How

Webhook subscriptions per project, fired from the same event pipeline that already writes to activityLogs — no new event system required.

⇢ One-Time Secret Sharing Links
Phase 2

For the rare case a single secret genuinely needs to leave the platform — replace the Slack-paste habit entirely, even for non-users.

What

Generate a link that reveals one secret exactly once, then self-destructs, optionally with an expiry window.

Why

Even great tooling can't prevent every ad-hoc "just send me the key" moment — this makes that moment safe instead of trying to eliminate it entirely.

How

Value is encrypted with a link-specific one-time key embedded only in the URL fragment (never sent to the server); the server marks the link consumed after first successful client-side decrypt.

Anything not listed here — and in particular anything that starts to resemble certificate management, dynamic infrastructure secrets, or agent/AI-credential governance — is a deliberate non-goal per the positioning in section 02, not a gap in planning.

15 Success Metrics

how we know MVP is actually done

  • Sign up and reach a working dashboard in under 1 minute (OAuth, no forms)
  • Create a project and import an existing .env in a single action
  • Pull a complete, working environment with one CLI command and zero global install
  • Push an update with a visible diff and one confirmation
  • Onboard a new teammate to a running local app without a single secret manually shared
  • See a meaningful environment-to-environment diff (dev vs. production) in under 5 seconds
★

North star: EnvVault becomes the single source of truth for environment variables for teams too small to need — or want — Vault, Infisical's full platform, or Doppler's enterprise tier.
